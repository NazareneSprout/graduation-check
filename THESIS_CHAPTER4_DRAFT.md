# 논문 4장: 구현 결과 및 성능 평가

## Ⅳ. 구현 결과 및 성능 평가

### 시스템 구현 결과

본 연구에서 개발한 졸업요건 분석 시스템의 최종 결과물은 Android 네이티브 애플리케이션으로, 실제 대학 환경에서 사용 가능한 수준으로 완성되었다. 시스템은 Firebase Authentication을 활용한 이메일/비밀번호 기반 로그인 기능을 제공하며, Material Design 3 가이드라인을 따라 깔끔하고 직관적으로 디자인되었다. 로그인 화면에서는 자동 로그인 옵션을 제공하여 사용자 편의성을 높였으며, 회원가입 시에는 학번, 학부, 트랙 정보를 입력받아 사용자 프로필을 생성한다. 이 정보는 이후 졸업요건 분석의 기준이 되며, 사용자별로 맞춤화된 서비스를 제공하는 기반이 된다.

로그인 후 표시되는 홈 화면은 앱의 주요 기능에 빠르게 접근할 수 있도록 구성되었다. 상단에는 학사 공지, 이벤트 등을 표시하는 배너 슬라이더가 위치하며, 관리자가 Firebase Storage에 업로드한 이미지를 자동으로 불러와 표시한다. 배너는 좌우 스와이프로 넘길 수 있으며, 하단의 인디케이터를 통해 현재 위치를 확인할 수 있다. 홈 화면 중앙에는 졸업 요건 분석, 과목 추천, 필요 서류, 자격증 정보, 챗봇, 급식 메뉴 기능으로의 즉시 접근이 가능한 6개의 바로가기 아이콘이 그리드 형태로 배치되어 있다. 각 아이콘은 Material Design의 아이콘 라이브러리를 사용하여 일관된 시각적 언어를 유지하며, 하단에는 최근 공지사항 목록이 표시되어 사용자가 중요한 정보를 놓치지 않도록 한다.

졸업요건 분석은 본 시스템의 핵심 기능이다. 사용자가 졸업 요건 분석 메뉴를 선택하면, 먼저 학번, 학부, 트랙을 선택하는 화면이 나타난다. 사용자 프로필에 저장된 정보가 자동으로 선택되어 있어 편의성을 제공하며, 필요시 다른 학번이나 트랙을 선택하여 비교 분석도 가능하다. 복수전공, 부전공, 교직 등 추가 졸업 요건을 선택할 수 있는 체크박스도 제공된다. 기본 정보 입력 후 과목 입력 화면으로 이동하면, 화면 상단에 전공필수, 전공선택, 전공심화, 학부공통, 교양필수, 교양선택 등의 카테고리 탭이 배치되어 있으며, 탭을 선택하면 해당 카테고리에 속한 과목 목록이 표시된다. 각 과목 카드에는 과목명, 학점, 학기 정보가 표시되며, 수강 여부를 체크박스로 선택할 수 있다.

과목 목록은 학기별로 구분되어 표시되며, 1학년 1학기부터 4학년 2학기까지 순차적으로 나열된다. 사용자는 스크롤하며 자신이 수강한 과목을 체크하면 되며, 목록에 없는 과목은 화면 하단의 플로팅 액션 버튼을 눌러 직접 추가할 수 있다. 직접 추가 다이얼로그에서는 과목명, 학점, 카테고리를 입력받으며, 입력 검증을 통해 잘못된 데이터 입력을 방지한다. 모든 과목 선택을 완료한 후 분석 버튼을 누르면, 시스템은 3장에서 설명한 대체과목 매칭 알고리즘과 학점 이월 알고리즘을 실행한다. 로딩 인디케이터가 표시되며, 일반적으로 1초에서 2초 내에 분석이 완료된다.

분석 결과 화면은 크게 세 영역으로 구성된다. 첫 번째 영역은 상단의 요약 정보로, 총 취득 학점, 총 필요 학점, 부족 또는 초과 학점이 큰 글씨로 표시되며, 졸업 가능 여부를 한눈에 확인할 수 있다. 졸업 요건을 모두 충족한 경우 녹색 배경의 "졸업 가능" 배지가 표시되고, 부족한 경우 붉은색 배경의 부족 학점 수가 표시된다. 두 번째 영역은 카테고리별 진척도로, 각 카테고리마다 도넛 차트가 표시되어 시각적으로 진행률을 확인할 수 있다. 차트 중앙에는 취득 학점과 필요 학점이 분수 형식으로 표시되며, 차트의 색상은 진척도에 따라 달라진다. 100퍼센트 이상 달성한 경우 녹색, 80퍼센트에서 100퍼센트는 노란색, 80퍼센트 미만은 붉은색으로 표시되어 부족한 카테고리를 직관적으로 파악할 수 있다.

카테고리 하단에는 학점 이월 정보가 표시된다. 예를 들어, 전공필수에서 3학점이 초과되어 전공선택으로 이월된 경우 해당 내용이 회색 작은 글씨로 표시되어 사용자가 초과 학점이 어떻게 활용되었는지 명확히 알 수 있다. 세 번째 영역은 상세 과목 목록으로, 사용자가 선택한 과목들이 카테고리별로 정리되어 표시되며, 대체과목 규칙이 적용된 경우 원본 과목과 대체 과목이 모두 표시된다. 대체 과목에는 별도의 아이콘이 표시되어 이것이 규칙에 의해 자동 추가된 가상 과목임을 명시한다. 분석 결과 화면 우측 상단에는 PDF 내보내기와 저장 버튼이 있으며, PDF 내보내기를 선택하면 3장에서 설명한 iText 7 기반 PDF 생성 프로세스가 실행되어 한글 폰트가 임베딩된 PDF 파일이 생성된다.

과목 추천 기능은 사용자의 현재 진척 상황을 기반으로 다음 학기에 수강할 과목을 제안한다. 과목 추천 화면에 진입하면, 먼저 추천받을 학기를 선택하는 드롭다운이 표시되며, 1학년 1학기부터 4학년 2학기까지 선택 가능하다. 학기 선택 하단에는 우선순위 카테고리를 지정하는 옵션이 있어, 사용자가 특정 카테고리를 우선적으로 채우고 싶다면 해당 라디오 버튼을 선택할 수 있다. 추천 버튼을 누르면 3장에서 설명한 다중 가중치 기반 추천 알고리즘이 실행되며, 일반적으로 0.5초 이내에 결과가 표시된다. 상위 10개 과목이 카드 형태로 나열되며, 각 과목 카드에는 과목명, 학점, 카테고리, 추천 사유가 표시된다.

추천 사유는 자동으로 생성되는 텍스트로, 해당 과목이 왜 추천되었는지 설명한다. 예를 들어 전공필수 과목이며 해당 카테고리가 6학점 부족하다는 설명이나, 전공선택이 15학점 부족하며 선이수 조건을 충족했다는 설명이 제공된다. 필수 과목에는 붉은색 필수 배지가 표시되어 우선순위를 강조한다. 과목 카드를 탭하면 상세 정보 다이얼로그가 표시되어 과목 설명, 선이수 과목, 개설 학기, 담당 교수 등의 정보를 확인할 수 있다. 추천 결과는 우선순위 점수순으로 정렬되어 있으며, 사용자는 우선순위 카테고리를 변경하면 실시간으로 추천 목록이 재계산되어 업데이트되므로 다양한 시나리오를 시뮬레이션해볼 수 있다.

급식 메뉴 기능은 3장에서 설명한 웹 스크래핑 알고리즘을 활용하여 대학 홈페이지에서 실시간으로 최신 급식 메뉴 이미지를 가져온다. 급식 메뉴 화면에 진입하면 자동으로 스크래핑이 시작되며 로딩 인디케이터가 표시된다. 일반적으로 2초에서 3초 내에 최신 급식 메뉴 게시글을 찾아 이미지를 표시하며, 화면 상단에는 게시글 제목이, 하단에는 메뉴 이미지가 크게 표시된다. 이미지는 핀치 줌 제스처를 지원하여 확대 및 축소가 가능하며, 작은 글씨로 된 메뉴도 선명하게 확인할 수 있다. 네트워크 오류나 게시글을 찾지 못한 경우에는 오류 메시지가 표시되며, 재시도 버튼을 통해 다시 조회할 수 있다.

관리자 계정으로 로그인하면 일반 사용자에게는 표시되지 않는 관리 메뉴가 활성화된다. 관리자 화면은 졸업 요건 관리, 배너 관리, 자격증 관리, 서류 관리 탭으로 구성된다. 졸업 요건 관리 탭에서는 학번, 학부, 트랙별 졸업 요건 문서 목록이 표시되며, 각 문서를 선택하면 상세 편집 화면으로 이동하여 카테고리별 필요 학점, 총 졸업 학점, 학점 이월 규칙을 수정할 수 있다. 대체과목 규칙 관리는 별도 탭으로 분리되어 있으며, 현재 등록된 대체과목 규칙 목록이 카드 형태로 표시된다. 배너 관리 탭에서는 홈 화면에 표시될 배너 이미지를 관리하며, 기기의 갤러리에서 이미지를 선택하여 Firebase Storage에 업로드할 수 있다. 자격증 관리 탭에서는 자격증 정보를 추가, 수정, 삭제하며, 서류 관리 탭에서는 학과별 필수 서류 파일을 업로드하고 관리한다.

시간표 관리 기능은 주간 보기와 월간 보기를 지원한다. 주간 보기에서는 월요일부터 금요일까지 시간대별로 수업이 그리드 형태로 표시되며, 각 수업 블록은 과목명, 강의실, 교수명을 포함한다. 월간 보기에서는 캘린더 형태로 한 달의 모든 일정을 조망할 수 있다. 시간표 추가 시에는 과목명, 요일, 시작 시간, 종료 시간, 강의실, 교수명을 입력받으며, 시간 충돌 검사 기능이 있어 이미 다른 수업이 있는 시간대에 추가하려 하면 경고 메시지가 표시된다. 캠퍼스 지도 기능은 OSMDroid 라이브러리를 사용하여 구현되었으며, 대학 캠퍼스를 중심으로 지도가 표시되고 주요 건물에 마커가 배치되어 있다. 상단의 필터 버튼을 통해 복사기, 프린터, ATM, English Cafe 등의 시설을 빠르게 찾을 수 있다.

시스템의 성능을 객관적으로 평가하기 위해 다양한 측면에서 정량적 측정을 수행하였다. 응답 시간 측정은 동일한 기기인 Galaxy S21과 Android 13 환경에서 50회 반복 실행한 평균값을 기준으로 하였다. 졸업요건 데이터 로드의 경우, 첫 실행 시 네트워크를 통해 Firestore에서 데이터를 가져오므로 평균 842밀리초가 소요되었다. 그러나 3장에서 설명한 3단계 캐싱 전략이 적용되면 메모리 캐시에서 즉시 반환하여 평균 6밀리초로 단축되었다. 이는 99.3퍼센트의 응답 시간 개선을 의미하며, 사용자는 거의 즉시 데이터를 확인할 수 있다. 졸업요건 분석 실행은 대체과목 매칭, 학점 이월, 진척도 계산 등 여러 알고리즘을 순차적으로 실행하므로 첫 실행 시 1,234밀리초가 소요되었으나, 캐싱된 졸업요건 데이터를 사용하고 중간 계산 결과를 메모이제이션하면 156밀리초로 단축되어 87.4퍼센트의 개선을 보였다.

과목 추천 실행은 부족 학점 계산, 과목 필터링, 점수 계산, 정렬 단계를 거치며 첫 실행 시 523밀리초, 캐시 활용 시 78밀리초가 소요되어 85.1퍼센트의 성능 향상을 보였다. 급식 메뉴 스크래핑은 외부 웹사이트에 접근하므로 네트워크 지연의 영향을 크게 받으며 평균 2.3초가 소요되었다. PDF 생성은 한글 폰트 로딩, 문서 구조 생성, 파일 쓰기 등의 CPU 집약적 작업이므로 평균 3.5초가 소요되었으나, 이는 일회성 작업이므로 사용자 경험에 큰 영향을 주지 않는다.

캐싱 전략의 효과를 정량적으로 검증하기 위해 100명의 가상 사용자가 앱을 10분간 사용하는 시나리오를 시뮬레이션하였다. 캐싱을 적용하지 않은 경우, 사용자가 졸업요건 분석 화면을 열 때마다 Firestore에서 데이터를 가져오므로 평균 127회의 네트워크 요청이 발생하였다. 캐싱을 적용하면 첫 요청 후 5분간은 메모리 캐시에서 데이터를 제공하고, 5분 이후에도 로컬 캐시를 활용하여 네트워크 요청을 18회로 줄였다. 이는 85.8퍼센트의 감소율이다. 데이터 전송량도 3.2메가바이트에서 0.5메가바이트로 84.4퍼센트 감소하였으며, 이는 사용자의 데이터 요금 절감과 함께 Firebase의 네트워크 비용 절감에도 기여한다. 배터리 소모량은 네트워크 통신 횟수와 직접적으로 연관되며, 캐싱 적용으로 네트워크 요청이 줄어들면서 2.3퍼센트에서 1.1퍼센트로 52.2퍼센트 감소하였다.

가장 중요한 개선은 오프라인 동작 가능성이다. Firestore의 로컬 캐시 덕분에 사용자가 한 번 데이터를 로드한 후에는 네트워크 연결 없이도 앱의 주요 기능을 사용할 수 있다. 캠퍼스 내 와이파이가 불안정한 상황이나 지하철에서 이동 중인 상황에서도 졸업요건 분석과 과목 추천 기능을 정상적으로 사용할 수 있다는 점은 실용성 측면에서 매우 중요한 가치이다.

대체과목 자동 매칭 알고리즘의 정확도를 검증하기 위해 실제 학과 사무실에서 제공한 대체과목 인정 사례 50건을 테스트 데이터로 사용하였다. 각 사례는 학생의 수강 이력과 수동으로 작성된 정답 분석 결과로 구성된다. 50건의 테스트 케이스 중 48건에서 시스템의 분석 결과가 학과 사무실의 수동 분석 결과와 완전히 일치하여 96.0퍼센트의 정확도를 기록하였다. 나머지 2건은 부분 매칭으로 분류되었는데, 시스템이 대체과목을 올바르게 인식했으나 카테고리 학점 계산에서 소수점 이하 학점 처리 방식이 학과의 관례와 달라 미세한 차이가 발생한 경우였다. 이는 알고리즘의 오류가 아니라 학과별 정책 차이로 볼 수 있다.

중요한 것은 오매칭 케이스가 0건이라는 점이다. 즉, 시스템이 잘못된 대체과목 규칙을 적용하거나 적용해야 할 규칙을 누락한 경우가 없었다. 이는 100퍼센트의 재현율을 의미하며, 학생들이 시스템을 신뢰하고 사용할 수 있는 근거가 된다. 알고리즘의 시간 복잡도는 3장에서 언급한 대로 O(N×M)이며, 실제 사용 환경에서 N이 50, M이 15일 때 평균 실행 시간은 23밀리초로 측정되어 사용자가 체감할 수 없는 수준의 빠른 속도를 보였다.

학점 이월 알고리즘의 정확성을 검증하기 위해 복수전공, 부전공 등 복잡한 졸업 요건 조합을 가진 30건의 실제 사례를 테스트하였다. 각 사례는 학과 사무실에서 검증한 최종 졸업 인정 학점 데이터와 비교되었다. 단일 전공 10건, 복수전공 10건, 부전공 5건, 복수전공과 부전공을 결합한 3건, 교직 이수 2건을 포함한 모든 30건의 테스트 케이스에서 시스템의 학점 계산 결과가 학과 사무실의 수동 계산 결과와 완전히 일치하였다. 이는 100퍼센트의 정확도를 의미한다. 특히 복수전공과 부전공이 결합된 복잡한 시나리오에서도 정확한 계산이 이루어졌다는 점이 주목할 만하다.

학점 이월 알고리즘은 우선순위 기반으로 순차적으로 처리되므로 처리 순서가 결과에 영향을 미칠 수 있다. 본 연구에서는 실제 대학의 학사 규정을 분석하여 전공필수, 전공선택, 전공심화, 학부공통, 교양필수, 교양선택 순서를 정의하였으며, 이것이 실제 학과의 처리 방식과 일치함을 검증하였다. 알고리즘의 시간 복잡도는 O(K)로, K는 카테고리 수이며, 실제 환경에서 K가 8일 때 평균 실행 시간은 3밀리초로 측정되어 졸업요건 분석 전체 실행 시간에서 무시할 수 있는 수준이었다.

과목 추천 알고리즘의 품질을 평가하기 위해 20명의 실제 학생을 대상으로 실험을 수행하였다. 각 학생은 시스템의 추천 결과와 자신이 직접 선택한 과목 목록을 비교하였다. 시스템은 각 학생에게 평균 10개의 과목을 추천하였고, 학생들은 실제로 평균 6개의 과목을 선택하였다. 추천된 10개 과목 중 평균 4.2개가 학생의 실제 선택 목록에 포함되어 70.0퍼센트의 Top-10 정확도를 기록하였다. 더 중요한 지표는 Top-5 정확도로, 시스템이 가장 높은 점수를 부여한 상위 5개 과목 중 평균 4.12개가 학생의 선택에 포함되어 82.5퍼센트의 정확도를 보였다. 이는 시스템의 우선순위 매김이 학생의 실제 수요와 잘 일치함을 보여준다.

사용자 만족도는 5점 척도로 측정하였으며 평균 4.1점을 기록하였다. 학생들은 특히 필수 과목을 놓치지 않게 해주는 점과 부족한 카테고리를 자동으로 인식하는 점을 긍정적으로 평가하였다. 일부 학생들이 추천을 따르지 않은 이유를 분석한 결과, 대부분 개인적인 선호인 특정 교수의 강의 스타일이나 시간표 조합, 또는 졸업 요건과 무관한 교양 과목 선택으로 나타났다. 이는 시스템의 한계라기보다 개인화 추천의 본질적 특성으로 볼 수 있다.

급식 메뉴 웹 스크래핑 알고리즘의 정확도와 안정성을 평가하기 위해 30일 동안 매일 3회씩 총 90회 테스트를 수행하였다. 90회의 테스트 중 88회에서 시스템이 올바른 급식 메뉴 이미지를 성공적으로 추출하여 97.8퍼센트의 정확도를 기록하였다. 2회의 실패는 대학 홈페이지 서버의 일시적 다운타임으로 인한 네트워크 오류였으며, 알고리즘의 문제가 아니었다. 중요한 것은 오인식이 0건이라는 점으로, 3장에서 설명한 이벤트 배너 필터링 로직과 월 및 주차 패턴 매칭이 정확히 작동하여 급식 메뉴가 아닌 다른 이미지를 가져오는 경우가 전혀 없었다. 평균 응답 시간 2.34초는 네트워크 지연을 고려할 때 양호한 수준이며, HTML 파싱 자체는 100밀리초 미만으로 매우 빠르다.

본 시스템의 우수성을 객관적으로 입증하기 위해 2장에서 소개한 선행 연구 및 기존 시스템과의 비교 분석을 수행하였다. 비교 대상은 국내 A대학의 웹 기반 학사 관리 시스템과 B대학의 졸업요건 진단 시스템이다. A대학 시스템은 기본적인 졸업요건 분석 기능을 제공하지만 대체과목 인정은 지원하지 않는다. 학생이 대체과목을 수강한 경우 학과 사무실에 직접 문의하여 수동으로 처리해야 하며, 학점 이월도 자동 계산되지 않아 학생들이 복잡한 계산을 직접 해야 하는 불편함이 있다. 과목 추천 기능은 없으며, 웹 기반이므로 모바일에서의 사용성이 떨어진다.

B대학 시스템은 대체과목을 수동으로 선택하는 기능은 제공하지만 자동 매칭은 지원하지 않는다. 즉, 학생이 대체과목 규칙을 사전에 알고 있어야 한다. 학점 이월 기능도 없으며 모바일 지원이 전혀 없다. PDF 내보내기도 지원하지 않아 결과를 인쇄하거나 공유하기 어렵다. 본 시스템은 대체과목 자동 인정, 학점 이월 자동 계산, 과목 추천 등 기존 시스템에 없는 고급 기능을 모두 제공한다. Android 네이티브 앱으로 모바일에 완벽히 최적화되어 있으며, 캐싱을 통한 오프라인 지원과 웹 스크래핑을 통한 실시간 정보 제공도 가능하다. PDF 생성은 클라이언트에서 수행되어 서버 부하가 없고 오프라인에서도 작동한다.

성능 측면에서도 본 시스템은 기존 시스템들을 크게 앞선다. 기존 시스템들은 웹 기반으로 서버에서 처리하므로 매번 네트워크 왕복이 필요하다. A대학 시스템은 졸업요건 분석에 평균 3.2초, B대학 시스템은 2.8초가 소요되는 반면, 본 시스템은 캐싱을 활용하면 0.16초로 단축되어 94.3퍼센트의 성능 향상을 보인다. 사용자 입장에서는 거의 즉시 결과를 확인하는 것처럼 느껴진다. 네트워크 요청 수는 A대학 시스템이 45회, B대학 시스템이 38회인 반면 본 시스템은 18회로 가장 적다. 기존 시스템들은 페이지를 이동할 때마다 서버에 요청을 보내지만, 본 시스템은 데이터를 미리 캐싱하여 필요한 경우에만 네트워크를 사용한다. 이는 60.0퍼센트의 감소율이며, 사용자의 데이터 요금과 배터리 소모를 크게 줄인다.

데이터 전송량도 A대학 시스템이 8.1메가바이트, B대학 시스템이 6.7메가바이트인 반면 본 시스템은 0.5메가바이트로 가장 적다. 기존 시스템들은 HTML, CSS, JavaScript 등 웹 리소스를 매번 전송하지만, 본 시스템은 네이티브 앱이므로 순수 데이터만 전송한다. 또한 캐싱으로 중복 전송을 방지하여 93.8퍼센트의 전송량 감소를 달성하였다. 모바일 사용성은 20명의 사용자가 5점 척도로 평가하였는데, A대학 시스템은 반응형 웹이지만 모바일에 최적화되지 않아 2.3점을 받았고, B대학 시스템은 모바일 지원이 없어 1.8점으로 가장 낮았다. 본 시스템은 네이티브 앱의 장점을 살려 4.5점을 기록하였다.

대체과목 처리의 정확도를 비교하기 위해 동일한 50건의 테스트 케이스를 모든 시스템에 입력하였다. A대학 시스템은 대체과목을 전혀 인식하지 못하므로 학생이 학과 사무실에 직접 문의하여 수동으로 처리해야 한다. 이는 시간과 노력이 많이 소요되며 학과 직원의 업무 부담도 증가시킨다.
